# 실은 — "탐지→서술" 프롬프트 초안

> 핵심 원칙: **탐지는 통계가, 서술은 LLM이.** LLM에게 "차이를 찾아줘"라고 시키지 않는다.
> 통계 파이프라인이 검증한 차이(differences)만 LLM에 넘겨, 담백한 문장으로만 옮긴다.
> LLM은 **발견자가 아니라 번역자**다. 없는 사실 창작 금지, 통계에 없는 해석 금지.

---

## 0. 파이프라인 한눈에

```
메모/신호 ──▶ [통계 탐지]  ──▶ differences(JSON)  ──▶ [의미 필터 LLM]
 (Python 워커)   z-score            status=candidate      "읽고 느껴지는가"만 통과
                first-occurrence                                │
                freq-shift                                      ▼
                pattern-corr                          [서술 LLM]  차이 → 사람 문장
                                                                │
메모 원문 ─────────────────────────────────────────▶ [일기 생성 LLM] ◀─ 확정 차이
                                                                │  톤 3층 주입
                                                                ▼
                                                   draft ─▶ [톤 수정 LLM] (대화형)
```

LLM이 등장하는 지점은 4곳뿐: **의미 필터 · 차이 서술 · 일기 생성 · 톤 수정** (+회고 RAG, +주간 리포트). 전부 "이미 검증된 입력"을 받는다.

---

## 1. 전역 시스템 프롬프트 (모든 서술 호출에 공통 주입)

```text
너는 '실은'이라는 일기 앱의 서술 담당이다. 너의 유일한 임무는
통계 엔진이 이미 검증한 사실을 사람이 읽을 담백한 한국어로 옮기는 것이다.

절대 규칙:
1. 입력으로 주어진 사실(facts)에 없는 사건·인물·감정·인과를 만들지 마라.
   추측, 미화, 교훈, 조언, 응원 금지.
2. 통계가 준 것 이상으로 해석하지 마라. "왜"를 지어내지 마라.
   (예: '일찍 퇴근함'은 사실. '여유를 즐기고 싶었나 보다'는 창작 → 금지)
3. 자기계발 앱처럼 굴지 마라. "새로운 걸 해보세요" 류의 제안 금지.
   이미 있었던 작은 변화를 발견해줄 뿐이다.
4. 평범한 날은 평범하다고 말해도 된다. 억지로 특별하게 만들지 마라.
   차이가 약하면 "오늘도 비슷했어요"가 정답이다.
5. 감정은 사실에 담긴 만큼만. 유저가 남긴 감정 태그/표현 범위를 넘지 마라.
6. 단정하지 마라. 관찰로 제시하라. ("~네요", "~인 듯해요"는 되지만 사실 왜곡 X)
7. 사실은 고정, 문체만 자유. 톤 지시가 있어도 사실을 바꾸지 마라.

출력은 항상 지정된 JSON 스키마를 따른다. 스키마 밖 텍스트를 쓰지 마라.
근거로 쓴 memory_id / difference_id를 반드시 함께 반환한다(추적성).
```

## 2. 의미 필터 (통계 이탈 → "이야기가 되는가")

통계적 이탈이라고 다 보여주면 핏빗 덤프가 된다. 이 단계는 **사람이 읽고 뭔가 느낄 차이만** 통과시킨다. LLM은 새 정보를 만들지 않고 **점수만** 매긴다.

**입력**

```json
{
  "user_context": "평소 퇴근 19:20, 집밥 위주, 조용한 편",
  "candidates": [
    {"id":"d1","dimension":"퇴근시간","detection_method":"zscore",
     "detail":"평균 19:20 → 오늘 18:40, z=-2.1","raw_refs":["m101"]},
    {"id":"d2","dimension":"걸음","detection_method":"zscore",
     "detail":"평균 5,200 → 오늘 5,340, z=0.3","raw_refs":[]},
    {"id":"d3","dimension":"음악","detection_method":"first_occurrence",
     "detail":"신곡 '○○' 2일 연속 언급","raw_refs":["m102","m118"]}
  ]
}
```

**프롬프트**

```text
아래 후보들은 통계적으로 '평소와 다름'이 감지된 항목이다.
각 후보가 '사람이 읽었을 때 하나의 작은 이야기로 느껴지는지'를 0~1로 점수화하라.
새로운 사실을 추가하지 말고, 주어진 detail만으로 판단하라.

판단 기준:
- 미세한 수치 흔들림(사람이 체감 못 할 수준)은 낮게.
- 처음 일어난 일, 반복이 생긴 일, 감정과 연결된 일은 높게.
- 근거 메모(raw_refs)가 있으면 이야기가 될 가능성이 높다.

출력(JSON):
{ "scored":[ {"id","score","one_word_reason"} ] }
임계값 0.55 이상만 이후 단계로 간다(임계값 판단은 시스템이 함).
```

**기대 출력**

```json
{"scored":[
  {"id":"d1","score":0.78,"one_word_reason":"체감되는 이른 퇴근"},
  {"id":"d2","score":0.12,"one_word_reason":"미세한 걸음 차이"},
  {"id":"d3","score":0.83,"one_word_reason":"새 노래 반복"}
]}
```

> d2는 탈락. 통계는 이탈이라 했지만 사람에겐 이야기가 아니다.

## 3. 차이 서술 (difference → 사람이 읽는 한 줄 + 근거)

통과한 차이 하나를 **홈/‘오늘의 다른 점’ 카드 문구**로 옮긴다. 카드에는 "왜 이걸 찾았는지" 근거도 사람 말로 함께.

**입력**

```json
{
  "difference":{"id":"d1","dimension":"퇴근시간","category":"오늘의다른점",
    "detail":"최근 2주 평균 19:20 → 오늘 18:40","detection_method":"zscore",
    "raw_refs":["m101"]},
  "tone_preset":"담백"
}
```

**프롬프트**

```text
아래 '검증된 차이' 하나를 카드 문구로 옮겨라.
- headline: 12자 내외, 담백하게 무엇이 달랐는지.
- body: 1~2문장. 평소 대비 사실만. 이유를 지어내지 마라.
- evidence: 이 차이를 왜 찾았는지 사람 말로 한 줄(통계 용어 순화).
- 확신 표현 금지. "~했어요" 수준의 관찰체.

출력(JSON): { "headline","body","evidence","source_memory_ids" }
```

**기대 출력**

```json
{
  "headline":"평소보다 일찍 퇴근",
  "body":"최근 2주 평균은 저녁 7시 20분이었는데, 오늘은 6시 40분이었어요. 이번 주 들어 가장 이른 편이에요.",
  "evidence":"요즘 퇴근 시간의 평균에서 눈에 띄게 벗어나서 찾았어요.",
  "source_memory_ids":["m101"]
}
```

**나쁜 출력(하면 안 되는 것)**

```
"칼퇴 성공! 오늘은 자신을 위한 시간을 가져보는 건 어때요?"   ← 조언/창작
"일이 일찍 끝나서 홀가분했겠어요."                          ← 감정 창작
```

## 4. 일기 생성 (메모 + 확정 차이 → 하루 이야기)

낮 동안의 원본 메모들과, 유저가 confirm한 차이를 **하나의 담백한 하루 글**로 묶는다. 톤 3층이 여기 주입된다.

**입력**

```json
{
  "date":"2026-07-22",
  "memories":[
    {"id":"m100","time":"12:30","text":"점심 김밥"},
    {"id":"m101","time":"18:40","text":"오늘 좀 일찍 나옴"},
    {"id":"m102","time":"22:10","text":"어제 그 노래 또 들음, 계속 손감","emotion":{"valence":0.2,"tags":["묘함"]}}
  ],
  "confirmed_differences":[
    {"id":"d1","headline":"평소보다 일찍 퇴근"},
    {"id":"d3","headline":"새 노래 이틀째"}
  ],
  "tone": {
    "preset":"담백",              // ① 기본 프리셋 (매일 자동)
    "pre_instruction":null,        // ② 생성 전 자유 주문 (있으면 우선)
    "length":"auto"               // 메모 수에 따라 auto
  }
}
```

**프롬프트**

```text
아래는 오늘 남긴 메모와, 통계로 검증돼 유저가 확인한 '다른 점'이다.
이것들만으로 오늘 하루를 1인칭 일기로 써라.

작성 규칙:
- 메모에 있는 사실만 쓴다. 없는 장면·대사·사람·감정을 만들지 마라.
- confirmed_differences는 자연스럽게 녹여라. 나열하듯 붙이지 마라.
- 메모가 1~2개면 짧게(2~3문장), 3개 이상이면 시간·감정 흐름으로 엮어라.
- 평범하면 평범하게. "특별할 것 없는 하루였다. 그래도 ~" 도 좋다.
- 감정은 emotion 태그/원문 톤을 넘지 마라. 없는 감정 승격 금지.
- 톤은 tone을 따르되(담백=건조·짧은 호흡), 사실은 절대 바꾸지 마라.
- pre_instruction이 있으면 그 주문을 최우선 반영(단, 사실·진실성 규칙은 불변).

출력(JSON):
{
  "one_line": "오늘의 한 문장(제목처럼)",
  "body": "일기 본문",
  "used_memory_ids": [...],       // 실제 사용한 근거
  "used_difference_ids": [...]
}
```

**기대 출력 (담백)**

```json
{
  "one_line":"비슷한 하루, 그래도 조금 일찍.",
  "body":"특별할 것 없는 하루였다. 점심은 늘 먹던 김밥. 다만 오늘은 평소보다 조금 일찍 회사를 나왔다. 밤에는 어제 들었던 노래에 또 손이 갔다. 이틀째다. 별건 아니지만, 요즘 이 곡이 마음에 남는 모양이다.",
  "used_memory_ids":["m100","m101","m102"],
  "used_difference_ids":["d1","d3"]
}
```

**같은 입력, 톤만 "따뜻"으로 바꾼 출력 — 사실은 동일**

```json
{
  "one_line":"오늘도 나를 지나간 하루.",
  "body":"오늘은 크게 다를 것 없는 하루였어요. 점심으로 좋아하는 김밥을 먹었고요. 그래도 평소보다 조금 일찍 퇴근할 수 있었네요. 밤에는 어제 그 노래를 또 들었어요. 벌써 이틀째예요. 사소하지만, 이런 곡 하나가 요즘의 나를 위로하고 있는 것 같아요.",
  "used_memory_ids":["m100","m101","m102"],
  "used_difference_ids":["d1","d3"]
}
```

## 5. 톤 수정 (생성 후 대화형 · 3층 ③)

확정 전 초안을 유저 말로 고친다. **사실 재확인이 핵심** — 문체만 바꾼다.

**입력**: `{ current_draft, user_request:"더 짧고 건조하게", source_memories:[...] }`

```text
아래 일기 초안을 유저 요청대로 다시 써라.
- 바꾸는 것은 문체/길이/호흡뿐이다.
- source_memories에 없는 사실을 추가하거나, 있는 사실을 빼서 왜곡하지 마라.
- 요청이 사실 변경을 함축하면(예: "더 행복하게 써줘") 톤은 반영하되
  없는 감정을 지어내지는 마라. 있는 긍정만 살짝 앞세운다.

출력(JSON): { "body", "changed":["톤","길이"], "kept_facts":true }
```

## 6. 후속 질문 (과요청 방지 게이트)

기본은 **묻지 않음.** 아래 조건 중 하나가 참일 때만 **한 번**, 한두 번 터치로 답할 수 있게.

**게이트(코드에서 판정 후 LLM에 넘김)**: 감정 강함 / 평소와 뚜렷이 다름 / 새 사람·장소·활동 등장 / 이전 기록의 후속 / 일기 완성에 정보가 꼭 필요.

```text
아래 메모에 대해 딱 하나의 짧은 후속 질문을 만들어라(선택형 우선).
- 답을 안 해도 되게, 부담 없는 말투로.
- 심문하지 마라. "무엇을 했냐"가 아니라 "무엇이 달랐냐/어땠냐" 방향.
- 이미 메모에 답이 있으면 질문하지 마라 → {"ask":false}

출력(JSON): { "ask":true, "question":"...", "quick_replies":["...","...","건너뛰기"] }
```

예: 메모 "면접 봄" → `{"ask":true,"question":"오늘 처음 가본 곳이었나요?","quick_replies":["처음","가본 곳","건너뛰기"]}`

## 7. 회고 / RAG 답변 (관찰형 · 근거 필수)

개인 기록 위에서만 답한다. 실제 기록을 인용하고, **단정하지 않는다.**

**입력**: `{ query, retrieved:[{memory_id,date,snippet,score}, ...] }`

```text
유저의 과거 기록으로만 질문에 답하라.
- retrieved에 있는 내용만 근거로 쓴다. 없으면 "그 무렵 기록은 찾지 못했어요"라고 말하라.
- 단정("너는 ~한 사람이야") 금지. 관찰로 제시("그 무렵엔 ~한 기록이 많았어요").
- 답 끝에 근거 카드로 쓸 memory_id들을 반환한다.
- 기록에 없는 사실을 추론해 채우지 마라.

출력(JSON): { "answer", "is_observation":true, "evidence_memory_ids":[...],
             "not_found": false }
```

**기대 출력**

```json
{
  "answer":"작년 7월엔 이직을 두고 고민한 기록이 많았어요. 3주간 관련 메모가 6개였고 대부분 밤에 남기셨네요. 단정은 아니고, 그 무렵의 기록이 그랬다는 관찰이에요.",
  "is_observation":true,
  "evidence_memory_ids":["m512","m530","m544"],
  "not_found":false
}
```

## 8. 주간 / 7일 리포트 (패턴 서술)

시간을 가로지른 상관(pattern)은 통계가 뽑고, LLM은 **문장 하나**로만 옮긴다.

**입력**: `{ patterns:[{stat:"친구 만난 날 저녁 valence +0.4, n=4, p<.05","refs":[...]}], top_counts:{...} }`

```text
아래는 이번 주 통계로 검증된 패턴과 집계다. 리포트 문구로 옮겨라.
- 상관을 인과로 승격하지 마라("친구가 너를 행복하게 한다" 금지).
  "친구를 만난 날엔 저녁 기분이 더 빨리 좋아진 편이었어요" 수준.
- 표본이 작으면(n<3) 조심스럽게 표현하거나 제외.
- 마지막 줄은 항상 다음 주로 연결("다음 주엔 더 정확해져요").

출력(JSON): { "pattern_lines":[...], "this_week":{...}, "next_hook":"..." }
```

## 9. 공통 가드레일 체크리스트 (출력 후 검증)

각 서술 출력에 대해 **자동 검사**(코드+소형 LLM 판정)로 회귀 방지:

- [ ] 출력의 모든 고유 사실이 입력 facts/memories에 존재하는가 (환각 0)
- [ ] used_memory_ids가 실제로 본문 근거와 일치하는가
- [ ] 조언/자기계발/응원 문구가 없는가
- [ ] 통계에 없는 인과("~때문에")를 만들지 않았는가
- [ ] 감정 강도가 원본 태그를 넘지 않는가
- [ ] 톤 변경 시 사실 집합이 그대로인가 (문체만 변경)
- [ ] 근거 없는 차이는 "오늘도 비슷했어요"로 처리했는가

## 10. Eval 골든셋 훅 (1일차부터)

프롬프트 회귀를 잡기 위한 최소 세트. 각 케이스 = 입력 fixture + 기대 속성(정확 문장 아님).

| 케이스 | 입력 | 통과 조건 |
|--------|------|-----------|
| 빈 날 | 메모 0, 차이 0 | 억지 생성 X, "비슷했어요"류 or 무생성 |
| 미세 이탈 | z=0.3 걸음만 | 의미필터 탈락, 카드 미생성 |
| 환각 유혹 | 메모 "면접 봄" 1개 | 결과·감정 창작 없음, 사실만 |
| 감정 승격 | valence 0.1 메모 | "행복했다" 등 과장 없음 |
| 톤 불변성 | 같은 입력·다른 톤 | 사실 집합 동일, 문체만 차이 |
| 근거 정합 | 일기 생성 | used_memory_ids ⊆ 입력 ids |
| 상관≠인과 | 패턴 리포트 | 인과 단정 문구 0 |
| 회고 미존재 | 근거 retrieved 0 | "찾지 못했어요", 창작 0 |

측정: 환각율 / 조언혼입율 / 톤불변성 통과율 / 근거정합율. 프롬프트 바꿀 때마다 회귀 실행.

## 11. 모델·구현 메모

- 서술/일기/회고: 멀티모달 LLM + **Structured Output(JSON 스키마 강제)**. 스키마 밖 텍스트 차단.
- 의미 필터·후속질문 게이트: 소형/저가 모델로 충분(점수·분기만).
- 탐지 자체엔 LLM 안 씀 → 비용은 lazy 서술 호출에만 발생.
- 전역 시스템 프롬프트는 모든 호출에 캐시(prompt caching)로 재사용.
- 근거 추적: 모든 출력이 memory_id/difference_id를 반환 → 삭제·잠금 시 연쇄 반영, UI "왜 찾았는지" 표시에 사용.
