# ADR-0002: 스키마 설계 리뷰 게이트 — 근거 연결·임베딩 무결성·삭제 완전성

- 상태: 채택
- 날짜: 2026-07-22
- 관련: [ADR-0001](./ADR-0001-stack-and-workflow.md), `.claude/rules/database.md`, `.claude/rules/privacy.md`, [설계 스펙](../superpowers/specs/2026-07-22-adr-0002-schema-gates-design.md), `docs/design/ERD.mermaid`

## 맥락 (Context)

`database.md`는 **첫 스키마 마이그레이션 전에** 네 가지를 명시 구조로 확정하라고 요구한다: 차이 근거 연결, 일기 출처, polymorphic 임베딩 무결성, 완전 삭제 추적. ADR-0001도 이를 후속 과제로 남겼다.

기존 `ERD.mermaid`는 이 게이트를 통과하지 못한 상태였다. `DIARIES.source_memory_ids`가 배열이고, `EMBEDDINGS`가 `(target_type, target_id)` 폴리모픽이며, 삭제 진행을 추적할 구조가 없었다.

지배적인 제약은 프라이버시다. `privacy.md`는 기억 삭제를 **원본 + Storage + 임베딩 + 검색 인덱스 + 파생 신호 전부**로 정의하고, 교차 사용자 노출을 심각 결함으로 규정한다. 폴리모픽 참조는 FK를 걸 수 없어 이 요구를 전적으로 애플리케이션 코드에 의존하게 만든다.

## 검토한 선택지 (Options)

**임베딩 구조**

1. **폴리모픽 유지 + 앱 레벨 검증** — 테이블 하나, 새 타입 추가가 쉬움 / FK가 없어 고아 벡터 방지가 코드 품질에 달림. 삭제 누락이 곧 프라이버시 사고.
2. **타입별 3테이블 + 검색 뷰** — FK CASCADE로 삭제 완전성이 DB 차원에서 보장 / 테이블이 늘고 교차 검색에 뷰가 필요.
3. 원본용 1테이블 + diary 1테이블 — 교차 검색이 뷰 없이 가능 / nullable FK 2개 + CHECK로 스키마가 덜 깔끔.

**삭제 유예기간**

1. **유예 없음, 즉시 완전 삭제** — 제품 약속과 일치, ledger 상태가 단순 / 오조작 복구 불가.
2. 30일 휴지통 — 복구 가능 / "지웠는데 남아있음"을 고지해야 하고, 유예 기간 내내 모든 조회 경로에 제외 필터가 필요. 한 곳만 놓쳐도 삭제한 기억이 일기에 등장.

**근거 메모가 일부 삭제된 DIFFERENCE**

1. 삭제하고 재생성 안 함 — 단순 / 사용자의 [맞아요] 판단까지 사라짐.
2. detector 재검증 후 갱신 — 통계라 비용이 낮음 / "지웠는데 또 뜸", status 리셋.
3. **무효화(stale)만 하고 재생성 안 함** — 판단 이력 보존, 과거 보존 / 행이 남아 파생값 처리를 따로 해야 함.

## 결정 (Decision)

상세 근거와 DDL은 [설계 스펙](../superpowers/specs/2026-07-22-adr-0002-schema-gates-design.md)에 있다. 결정만 옮기면:

1. **근거는 조인 테이블로** — `difference_evidence`, `diary_sources`. `DIARIES.source_memory_ids` 배열 제거. 메모에서 파생물을 역추적하는 인덱스를 함께 둔다(삭제 파이프라인의 핵심 질의).

2. **임베딩은 타입별 3테이블 + 원본 계열 UNION 뷰** (선택지 2). FK `NOT NULL` + `ON DELETE CASCADE`로 고아 벡터를 **구조적으로 불가능**하게 만드는 것이 결정적이었다. `user_id`와 `is_searchable`을 비정규화해 벡터 인덱스를 타면서도 user 스코프·잠금 필터를 강제한다. 잠금 동기화는 `memories → assets → asset_embeddings`까지 연쇄한다.

3. **유예기간 없는 즉시 완전 삭제 + deletion ledger.** `deleted_at`은 휴지통이 아니라 tombstone이며, ledger 완료 시 행 자체가 사라진다. 단계는 `search_index → derived → storage → db_hard_delete` 순이고, `storage`가 `db_hard_delete`보다 **앞서야** 한다(행을 먼저 지우면 `file_url`을 잃는다). `deletions.user_id`에는 `users` FK를 걸지 않는다 — 계정 삭제 중 ledger가 자신을 지우면 재개가 불가능해진다.

4. **파생물은 성격에 따라 다르게 처리한다.**
   - `BASELINES`는 **캐시** → 무효화 후 재구축. 남기면 지운 데이터의 영향이 평균에 계속 남는다.
   - `DIFFERENCES`는 **산출물** → 무효화(`evidence_state='stale'`)만 하고 재생성하지 않는다(선택지 3). `description`은 파기하고 사용자의 `status` 판단은 보존한다.
   - `DIARIES`는 **확정된 과거** → 불변.
   - `WEEKLY_REPORTS.highlights`는 스냅샷이라 방어선 밖에 사본을 만든다 → 조인 테이블 참조로 전환.

5. **동의 철회·기억 삭제로 일기가 영향받으면 사용자에게 묻는다.** 기본 선택은 "일기 남기기". 서비스가 임의로 지우지도 남기지도 않는다.

핵심 트레이드오프: **삭제 완전성을 위해 편의와 복구 가능성을 포기했다.** 개인 기록 제품에서 "지웠는데 남아있다"는 되돌릴 수 없는 신뢰 손상이고, 편의는 나중에 더할 수 있지만 유출된 데이터는 회수할 수 없다.

## 결과 (Consequences)

**긍정적**

- 고아 벡터·삭제 누락이 FK 제약으로 차단된다. 삭제 완전성이 코드 리뷰가 아니라 스키마의 책임이 된다.
- 근거 추적이 양방향으로 가능해져 "이 메모가 어디에 쓰였나"를 인덱스로 답할 수 있다.
- 부분 실패한 삭제를 ledger로 재개할 수 있다.
- 사용자의 [맞아요]/[아니에요] 판단이 삭제 연쇄에서 살아남는다.

**부정적 / 감수하는 비용**

- 삭제 오조작을 되돌릴 수 없다. UI에서 "되돌릴 수 없음"을 명시하고 내보내기 경로를 안내해야 한다.
- 남은 근거만으로도 성립했을 차이를 잃을 수 있다. "의무가 아니라 선물"이므로 놓쳐도 되는 범위로 본다.
- stale된 항목이 빠지면서 **과거 주간 리포트가 처음 봤을 때와 달라진다.** "과거는 그대로"와 엄밀히 어긋나지만, 지운 기억의 파생값을 남기는 것보다 낫다고 판단했다.
- 테이블이 늘어난다(임베딩 3 + 조인 3 + ledger 1). 교차 검색에 뷰가 필요하고 성능은 `EXPLAIN`으로 검증해야 한다.
- Next.js↔Python 두 자산이 같은 스키마를 공유하므로 마이그레이션 시 양쪽 배포 순서를 맞춰야 한다.

**후속 작업**

- 임베딩 차원 `N`과 모델 선택은 미정. 임베딩 구현(W6) 시점에 결정하고 필요하면 별도 ADR로 남긴다.
- `search_index` 단계의 실체(Postgres FTS인지 별도 엔진인지)는 하이브리드 검색 설계 시 확정한다.
- 완료된 ledger의 보존 기간은 운영 정책으로 정한다.
- 이 결정을 구현하려면 앱·워커 스캐폴딩이 선행되어야 한다(README "앱 스캐폴딩 (아직 미생성)").
